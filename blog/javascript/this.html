    this究竟是不是一种魔法？

    growydp 2/22/2022

    # 简单介绍一下

    this关键字是JavaScript中最复杂的机制之一，在严格模式和非严格模式之间有一点差别。

    绝大多数情况下，函数的调用方式决定了this的值（运行时绑定）。this不能再执行期间被赋值，并且在每次函数被调用时this的值也可能会不同

    正如《你不知道的JavaScript》中所说的，JavaScript中this的机制并没有那么先进，但是开发者往往会把理解过程复杂化。

    对于我来说，this真的让我头疼...我很讨厌这种模模糊糊的概念，之前也看过很多this相关文章，当时确实理解了，但是在第二天早晨就会陷入 马什么梅 什么冬梅 马冬什么？？？今天，我们将会彻底打败它！！！

    # 眼熟一下

    首先，从几个开发中经常能用到的例子开始展示this的魔法。

    # 初代（全局上下文）

    无论是否在严格模式下，在全局执行环境中（在任何函数体外部）this都指向全局对象。

    console.log(this === window) // true

    'use strict'
console.log(this === window) // true

    # 贰代（函数执行上下文之普通函数）

    例一

    var name = 'growydp'
function sayName() {
  console.log(name, this.name, this)
}
sayName()

    上面例子我们在开发中经常使用，自然能很快的说出打印内容。

    growydp growydp Window{}

    根据打印的第三个结果this输出Window全局对象来看，此时sayName函数的this指向Window。自然this.name就是取window.name。

    name则跟作用域链查找规则有关系，这里我们不过多赘述。

    例二

    现在我们简单修改一下代码

    var name = 'growydp'
function sayName() {
+ var name = 'tom'
  console.log(name, this.name, this)
}
sayName()

    我们再来看一下打印结果

    tom growydp Window{}

    跟我们上面分析的结论一样。这里需要注意的一点是，此代码运行在非严格模式下

    例三

    接下来，我们在函数内部顶部添加use strict;使其变为严格模式，然后打印一下this

    function sayName() {
  'use strict'
  console.log(this)
}
sayName()

    结果是undefined。也就是说，在严格模式下，如果进入执行环境时没有设置this的值，this会保持为undefined。

    # 仨代（函数执行上下文之对象的方法）

    当函数作为对象里的方法被调用时，this被设置为调用该函数的对象。

    例一

    var name = 'growydp'
var obj = {
  name: 'tom',
  sayName: function() {
    console.log(this.name)
  }
}

obj.sayName() // tom

    也可以先定义函数，然后再将其赋值给对象的方法

    function sayName() {
  console.log(this.name)
}
var name = 'growydp'
var obj = {
  name: 'tom',
  sayName
}

obj.sayName() // tom

    this的绑定只受最接近的成员引用的影响，比如上面的obj.sayName。我们将其扩充完整并输出结果

    window.obj.sayName() // tom

    也就是说，上面的代码中，sayName的this与obj有关系，与window没有关系。这是很重要的一点！！！

    例二

    本例子的坑我曾经踩过，害人不浅！！！与其说是坑，不过是粗心大意罢了~

    function sayName() {
  console.log(this.name)
}
var name = 'growydp'
var obj = {
  name: 'tom',
  sayName
}

var func = obj.sayName
func() // growydp

    上面仅仅是将obj对象的方法赋值给func了，但是没有调用！！！没有调用！！！没有调用！！！

    也可以这样理解，上面最后func函数调用等同于window.func()，按照上面例子得出的结论，方法最近的是window，结果就是growydp。（不知道我这样理解对不对）

    # 小总结

    this是在运行时进行绑定的，不是在编写时绑定，它的上下文取决于函数调用时的各种条件。this的绑定和函数声明的位置没有任何关系，只取决于函数的调用方式。

    当一个函数被调用时，会创建一个活动记录（执行上下文）。这个记录会包含函数在哪里被调用（调用栈）、函数的调用方式、传入的参数等信息。

    # 叛逆的this

    凭什么调用我的位置在全局执行上下文，我的this就要指向Window？？？我偏不！！！接下来，我们将帮助叛逆的this改变其指向。

    # call/apply/bind

    JavaScript提供的绝大多数函数以及自己创建的所有函数都可以使用call(...)和apply(...)方法。

    他们的第一个参数是一个对象，是给this准备的，其余参数：

    • apply第二个参数接收一个数组
    • call接收多个参数列表

    例一

    var obj = {
  age: 25
}

function sayName() {
  console.log(this.age)
}

sayName.call(obj) // 25

    如果直接调用sayAge函数将会返回undefined

    如果传入一个原始值（字符串类型、布尔类型或者数字类型）来当做this的绑定对象，这个原始值会被转换成它的对象形式（new String() new Boolean() new Number()）。

    例二

    function sayName() {
  console.log(this.age)
}

var obj = {
  age: 20
}

var _bind = function() {
  sayName.call(obj)
}

_bind() // 20
setTimeout(_bind, 1200) // 20

    上面代码使用了强绑定，也就是即使给_bind指定this也不会影响_bind函数内给sayName绑定obj

    ES5提供Function.prototype.bind

    function sayAge(num) {
  console.log(this.age, num)
  return this.age + num
}

var obj = {
  age: 20
}

var age = sayAge.bind(obj)
var add = age(3)
console.log(add)

// 20, 3
// 23

    bind会返回一个硬编码的新函数，会把指定的参数设置为this的上下文并调用原始函数

    # new绑定

    JavaScript中的“构造函数”只是一些使用new操作符时被调用的函数。他们不会属于某个类，也不会实例化一个类。

    function Person(age) {
  console.log(this) // Person {}
  this.age = age
}

const p = new Person(20)
console.log(p) // Person {age: 20}
console.log(p.age) // 20

    扩展：面试我们经常遇到面试官让我们手写new，接下来记住以下步骤：

    1. 创建一个全新的对象
    2. 回想实例和构造函数原型的关系，即instance.__proto__ = constructor.prototype
    3. 将新创建的对象设置为构造函数的this，这样在构造函数中写this.age = age即可给这个对象赋值
    4. 如果函数没有返回其他对象，那么new表达式中的函数调用会自动返回这个新对象
    function _new(fn, ...args) {
  const obj = Object.create(null)
  Object.setPrototypeOf(obj, fn.prototype)
  fn.apply(obj, args)
  return obj
}

    上面在实现 new 的时候我使用Object.create(null)创建了一个对象，为什么不直接使用{}创建呢？

    因为Object.create(null)不会创建Object.prototype这个委托，不信？上代码

    const o = Object.create(null)
console.log(Object.getPrototypeOf(o)) // null
const o1 = {}
console.log(Object.getPrototypeOf(o1)) // {constructor: f .....}

    总结一点，使用Object.create(null)创建的对象很“纯净”

    使用new来调用foo()时，会构造一个新对象并把它绑定到foo()调用中的this上

    # 优先级

    某个调用位置应用多条规则的情况下，就需要给这些规则设定优先级。

    # 隐式绑定和显式绑定

    function sayAge() {
  console.log(this.age)
}

const person1 = {
  age: 22,
  sayAge
}

const person2 = {
  age: 30,
  sayAge
}

// 隐式绑定
person1.sayAge() // 22
person2.sayAge() // 30

// 显式绑定
person1.sayAge.call(person2) // 30
person2.sayAge.call(person1) // 20

    显式绑定的优先级高于隐式绑定。

    # new绑定和隐式绑定

    function Person(age) {
  this.age = age
}

const o1 = {
  Person
}

const o2 = {

}

// 隐式绑定
o1.Person(20)
console.log(o1.age) // 20

o1.Person.call(o2, 30)
console.log(o2.age) // 30

// new绑定
const p = new o1.Person(40)
console.log(o1.age) // 20
console.log(p.age) // 40

    new绑定比隐式绑定优先级高

    因为new无法与显式绑定的call/apply一起使用。但是可以使用硬绑定来测试优先级

    # 判断this

    1. 函数是否在new中调用(new 绑定)？如果是的话this绑定的是新创建的对象。
    const p = new Person()
    1. 函数是否通过call/apply（显式绑定）或者硬绑定调用？如果是的话，this绑定的是指定的对象。
    const age = sayAge.call(o1)
    1. 函数是否在某个上下文对象中调用（隐式绑定）？如果是的话，this绑定的是那个上下文对象
    const age = o1.sayAge()
    1. 如果都不是，就是默认绑定。在严格模式下，就绑定到undefined，否则绑定到全局对象
    const age = sayAge()

    # 绑定特殊情况

    # null和undefined传入显示绑定

    当我们视图把null或者undefined作为this的绑定对象传入call/apply/bind。这些值在调用时会被忽略，实际应用的是默认规则

    function func(...rest) {
  console.log(rest)
}

const arr = [1, 2, 3, 4, 5, 6, 7]

func.apply(null, arr)

// [1, 2, 3, 4, 5, 6, 7]

    # 安全的this

    function func(...rest) {
  console.log(rest)
}

const o = Object.create(null)

const arr = [1, 2, 3, 4, 5, 6, 7]

func.apply(o, arr)

    上面我们给func函数指定了纯净的对象o，比null的含义更清楚
